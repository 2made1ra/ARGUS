# Unified Chat, Search And UI Plan

## Goal

Make one chat the primary user interface. The assistant decides internally whether to gather a brief, search suppliers/items, do both, or ask clarification. The UI shows the chat, current brief draft and found catalog items together.

## Product Model

Do not ask users to choose a tab before they understand their task. The same message can be brief discovery, supplier search or both:

```text
"Хочу музыкальный вечер" -> brief_discovery
"Мне нужно музыкальное оборудование на вечер в концертный зал" -> supplier_search
"Хочу организовать музыкальный вечер на 100 человек, помоги понять что нужно" -> mixed
```

The interface is one workflow:

```text
user message
  -> assistant router
  -> BriefState update when useful
  -> search_items when useful
  -> response message
  -> visible brief draft
  -> visible found item cards
```

## Assistant Response Contract

The assistant response has three separate layers. MVP must not collapse them into one prose answer:

```text
message
  -> live human explanation, grouping, clarifying questions and next step

brief
  -> structured BriefState for the event

found_items
  -> checkable price_items rows from Postgres
```

Contract rules:

- `message` is not the source of truth for catalog facts.
- `found_items` is the primary search result when the user asks for services, goods, suppliers or prices.
- `brief` is the structured event state; a prose brief may be rendered from it, but MVP does not require final proposal generation.
- The UI must visually separate assistant prose, "Черновик брифа" and "Найденные позиции".
- All concrete claims about `unit_price`, `unit`, `supplier`, `supplier_city`, `category`, phone, email, INN or availability must be backed by a row in `found_items` or an opened catalog item detail.
- If no catalog row backs a claim, the assistant must phrase it as an assumption, recommendation or missing data.
- Search results should be shown as cards/table before or alongside the explanation, not hidden inside a text summary.

Minimum `found_items` card fields:

```text
name
unit_price
unit
supplier
supplier_city
category
source_text_snippet
source_text_full_available
match_reason
```

`match_reason` is generated by the backend search service, not by free-form LLM prose. Use safe reason codes or templated explanations such as `semantic`, `keyword_name`, `keyword_supplier`, `keyword_inn` and `keyword_source_text`. The assistant can group or explain results, but it must not invent why a row matched.

## Assistant Style Rules

- First acknowledge the user's task in plain language.
- Then state what is already known from the user or `BriefState`.
- Separate assumptions from facts found in the catalog.
- Ask at most 3-5 high-value clarification questions when data is missing.
- If broad search is useful, show preliminary `found_items` and ask for missing filters in parallel.
- Do not say a supplier or item is "ideal" unless the data explicitly supports that claim.
- Do not promise date availability, stock, booking status or final budget when the CSV/catalog has no such fields.
- Do not invent suppliers, prices, phones, emails, INNs or cities.
- For empty results, say that the catalog has no matching rows yet and suggest how to refine the query.

## Router Contract

Use structured output or a strict parser around the local LLM response. The assistant use case must normalize every turn into:

```json
{
  "intent": "brief_discovery",
  "confidence": 0.86,
  "known_facts": {
    "event_type": "музыкальный вечер"
  },
  "missing_fields": ["city", "audience_size", "venue"],
  "should_search_now": false,
  "search_query": null,
  "brief_update": {
    "event_type": "музыкальный вечер"
  }
}
```

Allowed intents:

```text
brief_discovery | supplier_search | mixed | clarification
```

Rules:

- `brief_discovery`: update brief and ask up to 3-5 important questions; do not search if the query is too abstract.
- `supplier_search`: form a concrete search query and call `search_items`; ask for missing filters without blocking broad search.
- `mixed`: update brief and call `search_items` for obvious required services.
- `clarification`: ask a focused follow-up when confidence is low or the message cannot be routed safely.

## Brief State Contract

MVP stores one active brief per chat/session:

```json
{
  "event_type": null,
  "city": null,
  "date_or_period": null,
  "audience_size": null,
  "venue": null,
  "venue_status": null,
  "duration_or_time_window": null,
  "budget": null,
  "event_level": null,
  "required_services": [],
  "constraints": [],
  "preferences": []
}
```

Rules:

- Treat brief as state, not as a prose-only chat summary.
- Merge new user facts into the existing brief.
- Preserve unknown values as `null` or empty arrays.
- Do not build complex brief history in MVP.
- Use `venue_status` for "площадка есть" vs "площадку нужно подобрать".
- Use `duration_or_time_window` for duration or rough schedule when the user provides it.
- Use `event_level` for coarse positioning such as `базовый`, `средний` or `премиум`; if confidence is low, keep it in `preferences` instead of forcing a value.

## Backend Tools

### `search_items`

Input:

```json
{
  "query": "акустическая система для концертного зала",
  "filters": {
    "supplier_city": "г. Москва",
    "category": ["Оборудование", "Аренда"],
    "supplier_status": "Активен"
  },
  "limit": 10
}
```

Output:

```json
{
  "items": [
    {
      "id": "uuid",
      "score": 0.82,
      "name": "Аренда акустической системы",
      "category": "Аренда",
      "unit": "день",
      "unit_price": "15000.00",
      "supplier": "ООО Пример",
      "supplier_city": "г. Москва",
      "source_text_snippet": "фрагмент исходной строки",
      "source_text_full_available": true,
      "match_reason": {
        "code": "semantic",
        "label": "Семантическое совпадение с запросом про акустическую систему"
      }
    }
  ]
}
```

`search_items` MVP behavior:

```text
1. Semantic search in Qdrant using the new catalog embedding.
2. Minimal Postgres keyword fallback over name, source_text, supplier, supplier_inn and external_id.
3. Merge and dedupe by price_item_id.
4. Hydrate all returned rows from Postgres.
5. Return backend-generated match_reason and source_text_snippet.
```

This is not full hybrid search. Sparse vectors, RRF, trigram tuning and quality-metric-driven reranking remain post-MVP.

### `POST /assistant/chat`

The primary user endpoint.

Response semantics:

- `message`: live assistant prose for explanation, grouping, clarification and next step.
- `router`: structured routing decision used for observability/debugging.
- `brief`: current structured event state after deterministic merge.
- `found_items`: checkable catalog rows; this is the primary result of supplier/item search.
- Do not add new MVP fields such as `answer_sections`; keep the contract simple and make the UI separation explicit.

Request:

```json
{
  "session_id": "uuid-or-null",
  "message": "Хочу организовать музыкальный вечер на 100 человек",
```

## Frontend UI Contract

First screen layout:

```text
---------------------------------------------------------
| Chat thread                         | Brief draft      |
| User/assistant messages             | Found items      |
| Composer                            |                  |
---------------------------------------------------------
```

Rules:

- Do not render primary user tabs for "поиск" vs "бриф".
- Keep admin/debug screens separate: CSV upload, import history, embedding/search debug.
- Render assistant prose separately from the brief and item table/cards.
- Brief draft is visible and updates after each assistant turn.
- Found items are visible as cards or a dense table.
- Each found item shows `name`, `unit_price`, `unit`, `supplier`, `supplier_city`, `category`, `source_text_snippet` and backend-generated `match_reason`.
- Item click opens `/catalog/items/:id`.
- Do not show RAG prose as the primary answer. Assistant prose can explain what it did, but the factual result is the item cards/table.
- A future prose event brief can be displayed as a separate block, but it must be generated from `BriefState` and selected/found catalog rows.
- Treat `found_items` as candidates. A later selection workflow should add `selected_item_ids` or "Добавить в подборку" before final proposal generation, so the assistant does not treat every found candidate as chosen.

# Event Brief Assistant Orchestrator Phase Prompts

Готовые промпты для пофазного выполнения
[`07-event-brief-assistant-orchestrator.md`](../catalog-first-refactor/07-event-brief-assistant-orchestrator.md).

Промпты составлены под `gpt-5.5` по принципам OpenAI Prompt Guidance:

- начинать с желаемого результата и критериев успеха;
- держать стабильный контекст и правила выше переменной задачи;
- задавать явные границы, stop conditions и формат финального ответа;
- использовать структурированные контракты там, где агент должен вернуть DTO или план действий;
- дробить работу на небольшие проверяемые фазы вместо одного большого PR;
- не просить модель раскрывать скрытые рассуждения, а требовать короткое техническое обоснование решений.

Источник: <https://developers.openai.com/api/docs/guides/prompt-guidance>

## Как Использовать

1. Используй один промпт на одну сессию, ветку или PR.
2. Для первого backend-среза используй `MVS-1 Prompt`; для более мелкой работы используй фазовые промпты ниже.
3. Не проси агента переходить к следующей фазе, пока текущая не реализована и не проверена.
4. Для сложных backend-фаз используй `gpt-5.5` с повышенным reasoning effort; для docs-only или contract-only правок достаточно среднего reasoning effort.
5. Не вставляй весь план 07 в промпт: агент должен прочитать файл из репозитория.

## Общий Блок Для Любой Фазы

Добавляй этот блок в начало фазового промпта, если агент не видит текущий контекст репозитория:

```text
Ты работаешь в репозитории ARGUS.

Goal:
Выполни только указанную фазу плана event-brief assistant orchestrator. Цель продукта:
ARGUS assistant is an event-brief copilot with catalog-backed supplier search, not a free-running agent.

Read first:
- CLAUDE.md
- AGENTS.md
- docs/agent/architecture.md
- docs/agent/search.md
- docs/agent/entity-resolution.md
- docs/plans/catalog-first-refactor/07-event-brief-assistant-orchestrator.md
- docs/plans/catalog-first-refactor/08-event-brief-e2e-scenario.md

Non-negotiable constraints:
- Business logic belongs in feature use cases/domain services, never FastAPI routes or adapters.
- price_items in Postgres is the source of truth for catalog facts.
- Qdrant price_items_search_v1 is the controlled vector index for catalog search.
- Assistant prose must not invent suppliers, prices, cities, INNs, contacts, statuses or availability.
- Catalog facts come only from found_items, item details, selected items or verification_results.
- LLM output may help structured interpretation only; it must not authorize tools or generate unchecked final facts.
- One chat turn is bounded: no autonomous infinite loop, no arbitrary tools, no SQL/HTTP generated by the LLM.
- Preserve existing MVP behavior unless the phase explicitly changes it.
- Do not implement later phases in this session.
- Do not commit, push or open a PR unless explicitly asked.

Implementation style:
- Use TDD for behavior changes: write focused failing tests first, then implementation.
- Prefer small DTO/domain services over logic inside routes or adapters.
- If current code conflicts with the plan, stop and explain the conflict before broad edits.
- Keep response layers separate: message, router/action diagnostics, brief, found_items, verification_results, rendered_brief.

Final response format:
- Changed files grouped by layer.
- Verification commands run and exact results.
- Skipped checks or blockers.
- Any behavior intentionally left for a later phase.
```

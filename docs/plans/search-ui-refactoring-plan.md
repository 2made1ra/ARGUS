# План рефакторинга глобального AI-поиска

Дата: 2026-05-03

Статус: draft

## Контекст

Текущий экран AI-поиска выглядит как демонстрация RAG-механики, а не как рабочий инструмент ARGUS. В один экран сейчас смешаны три разные задачи:

- чат-ответ LLM;
- ранжирование подрядчиков;
- проверка источников и найденных фрагментов.

Для ARGUS основной сценарий уже задан проще и сильнее:

```text
тема -> подрядчик -> документы -> фрагмент
```

Поэтому глобальный поиск должен отвечать прежде всего на вопрос: **кого выбрать и почему**. Чат и подробная работа с фрагментами должны появляться глубже: на уровне подрядчика, документа или конкретного найденного фрагмента.

## Наблюдения по текущему интерфейсу

Проблемы текущего экрана:

- поле ввода находится ниже ответа, хотя поиск должен начинаться сверху;
- `RagChat` показывает источники, а правая панель повторно показывает подрядчиков и те же источники;
- raw score вида `0.777` выводит внутреннюю механику ранжирования вместо пользовательской оценки;
- источники иногда отображаются как `стр. ?`, что снижает доверие к результату;
- markdown из LLM протекает в UI как `**Причина:**`;
- подрядчики и фрагменты источников конкурируют за внимание;
- крупный hero-заголовок и две панели делают экран визуально тяжелее самой задачи.

Ключевой вывод: нужно упростить экран концептуально, а не только косметически.

## Внешние UX/RAG-паттерны

Паттерны, которые стоит использовать в ARGUS:

- Ответ должен быть проверяемым: inline citations и список источников со snippet, title и ссылкой на источник.
- Источники должны вести к точному месту: документ, страница, фрагмент или deep link, если backend это поддерживает.
- Пользователь должен уметь сужать область поиска: подрядчик, документ, тип документа, дата, сумма, статус.
- Сложность retrieval должна быть скрыта от пользователя: query understanding, reranking, chunk selection и hybrid search не должны становиться элементами интерфейса.
- Фильтры стоит показывать как примененные чипы после поиска, а не перегружать ими первый экран до запроса.

Референсы:

- [Amazon Q Business: source attribution and citations](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/source-attribution-citations.html)
- [Glean: citations](https://docs.glean.com/user-guide/assistant/glean-chat/glean-chat-citations/glean-citations)
- [Glean: deep-linked citations](https://docs.glean.com/user-guide/assistant/glean-chat/glean-chat-citations/deep-linked-citations)
- [NotebookLM: sources](https://support.google.com/notebooklm/answer/16215270?co=GENIE.Platform%3DDesktop&hl=en-GB)
- [Azure AI Search: RAG overview](https://learn.microsoft.com/en-us/azure/search/retrieval-augmented-generation-overview)
- [Baymard: product list and filtering](https://baymard.com/blog/current-state-product-list-and-filtering)
- [Baymard: applied filters](https://baymard.com/blog/how-to-design-applied-filters)
- [Qdrant: hybrid queries](https://qdrant.tech/documentation/concepts/hybrid-queries/)
- [Pinecone: sparse and hybrid search](https://docs.pinecone.io/guides/data/encode-sparse-vectors)

## Целевая модель экрана

Глобальный поиск должен быть одним вертикальным потоком:

```text
[ Найти подрядчика, условие договора или риск... ] [Найти]

Короткий вывод ARGUS
ООО «МАЛИНБУ-СПОРТ» выглядит наиболее подходящим подрядчиком.
Основание: организация мероприятий, стоимость, обязанности сторон. [S1] [S2]

Подходящие подрядчики
1. ООО «МАЛИНБУ-СПОРТ»      Высокое совпадение
   Почему подходит: ...
   2 договора · 5 фрагментов · Открыть подрядчика

2. ...

Источники
[S1] Договор ... · стр. 1-2 · фрагмент
[S2] Договор ... · стр. 2-3 · фрагмент
```

Постоянная правая панель не нужна. Preview источника можно добавить позже как drawer или side panel, который открывается только по клику.

## Этап 1. Упростить глобальный экран поиска

Цель: превратить экран из RAG-демо в рабочий поиск подрядчиков.

Затрагиваемые файлы:

- `frontend/src/pages/SearchPage.tsx`
- `frontend/src/components/RagChat.tsx`
- `frontend/src/components/SourceList.tsx`
- `frontend/src/App.css`

Изменения:

- убрать постоянную правую колонку `GlobalResultMirror`;
- оставить один основной поток: поле поиска -> короткий ответ -> подрядчики -> источники;
- перенести поле ввода наверх экрана;
- заменить огромный hero-заголовок компактным заголовком рабочего инструмента;
- показывать источники один раз, а не в нескольких местах.

Ожидаемый результат:

- пользователь сразу видит поле поиска;
- результаты читаются сверху вниз;
- экран перестает дублировать один и тот же смысл.

## Этап 2. Очистить представление ответа

Цель: убрать технические артефакты из интерфейса.

Изменения:

- скрыть raw score вида `0.777`;
- заменить score на уровни `Высокое совпадение`, `Среднее совпадение`, `Низкое совпадение`;
- исправить отображение markdown в ответе;
- не показывать `стр. ?` как нормальное состояние;
- для неизвестной страницы выводить `страница не распознана`;
- укоротить длинные имена PDF, чтобы карточки не распирало.

Ожидаемый результат:

- интерфейс выглядит как продукт, а не как отладочная панель;
- пользователь видит понятную оценку совпадения;
- проблема отсутствующих страниц становится явной.

## Этап 3. Разделить глобальный поиск и чат

Цель: не смешивать разные режимы работы.

### План выполнения

Этап 3 выполняется как frontend-only рефакторинг без изменения backend/API.
Экран `/search` перестает использовать `RagChat` и получает отдельный компонент
`GlobalSearchAnswer`, который работает как поисковый интерфейс выбора подрядчика,
а не как чат с историей сообщений.

`RagChat` остается контекстным инструментом только для страниц подрядчика и
документа.

Ключевые изменения:

- создать `frontend/src/components/GlobalSearchAnswer.tsx`;
- хранить внутри компонента `input`, `latestAnswer`, `loading`, `error`;
- отправлять запрос через `answerGlobalSearch(question, [])`;
- не накапливать `ChatMessage[]` и не показывать user/assistant bubbles;
- рендерить поток: форма поиска -> короткий вывод ARGUS -> подрядчики -> источники;
- использовать `AssistantContent`, `SourceList` и `matchLabel`;
- обновить `frontend/src/pages/SearchPage.tsx`, оставив там только layout страницы и
  `<GlobalSearchAnswer />`;
- не менять `api.ts`, DTO и endpoint `/search/answer`;
- оставить `RagChat` для `ContractorPage` и `DocumentPage` без изменения поведения.

Целевая логика экранов остается:

```text
Глобальный поиск
-> список подходящих подрядчиков
-> профиль подрядчика
-> документы подрядчика
-> поиск или чат внутри конкретного документа
```

Ожидаемый результат:

- глобальный экран отвечает на вопрос выбора подрядчика;
- чат становится контекстным инструментом, а не основной формой глобального поиска.

Критерии проверки:

- `npm run build` в `frontend` проходит успешно;
- `SearchPage.tsx` больше не содержит `<RagChat`;
- `<RagChat` остается только на страницах подрядчика и документа;
- `GlobalSearchAnswer.tsx` не использует `chat-bubble` и `chat-thread`;
- на `/search` после запроса нет истории сообщений и пользовательского bubble;
- порядок результата: короткий вывод ARGUS -> подходящие подрядчики -> источники;
- источники отображаются один раз;
- длинные названия и карточки не создают горизонтальный скролл на mobile.

## Этап 4. Улучшить карточки подрядчиков

Цель: сделать подрядчика главным объектом выбора.

### План выполнения

Этап 4 выполняется как frontend-only улучшение `GlobalSearchAnswer`.
Подрядчик становится главным объектом выбора, а источники внутри карточек
показываются как короткие доказательства, связанные с этим подрядчиком.

Backend/API/DTO не меняются. Используются уже доступные
`latestAnswer.contractors` и `latestAnswer.sources`.

Ключевые изменения:

- обновить `frontend/src/components/GlobalSearchAnswer.tsx`;
- заменить простые `Link.result-card` на карточку `<article>` без вложенных
  ссылок;
- оставить кликабельное действие `Открыть подрядчика` как отдельный `Link` на
  `/contractors/:id`;
- показывать название, текстовый уровень совпадения, `top_snippet`, количество
  документов и найденных фрагментов;
- добавить подпись `Почему подходит` перед `top_snippet`;
- добавить блок `Ключевые источники` внутри карточки;
- для каждой карточки брать первые 1-2 `SourceRef` из `answer.sources`, где
  `source.contractor_id === contractor.contractor_id`;
- если `contractor_id` у источника отсутствует, не подмешивать его в карточку;
- глобальный блок `Источники` внизу оставить без изменений как полный список
  фрагментов;
- источник внутри карточки вести на `/documents/:document_id`;
- показывать `S{globalIndex}`, короткое название документа через
  `compactDocumentTitle`, страницу через `formatPageRange` и snippet источника;
- не показывать raw score.

Стили:

- добавить классы `.contractor-card`, `.contractor-card__top`,
  `.contractor-card__reason`, `.contractor-card__stats`,
  `.contractor-card__actions`, `.contractor-card__sources`;
- визуально отделить карточки подрядчиков от нижнего списка источников:
  карточки крупнее, с CTA и evidence-блоком;
- глобальные источники оставить компактными chips;
- сохранить mobile-safe правила: `min-width: 0`, `overflow-wrap: anywhere`,
  line clamp для названий и snippet;
- не делать карточку целиком ссылкой, потому что внутри есть отдельные ссылки на
  документы-источники.

Критерии проверки:

- `npm run build` в `frontend` проходит успешно;
- `rg "toFixed\\(3\\)|score " frontend/src/components/GlobalSearchAnswer.tsx`
  не показывает пользовательский raw score;
- `rg "contractor-card" frontend/src/components/GlobalSearchAnswer.tsx frontend/src/App.css`
  подтверждает новые классы;
- `rg "<Link[^>]*className=\"result-card" frontend/src/components/GlobalSearchAnswer.tsx`
  не находит старую карточку-ссылку;
- на `/search` карточка показывает название подрядчика, причину, счетчики и
  `Открыть подрядчика`;
- внутри карточки есть 1-2 ключевых источника, если они есть для этого
  подрядчика;
- нижний блок `Источники` остается полным списком и визуально отличается от
  карточек;
- длинные названия подрядчиков/PDF не создают горизонтальный скролл на mobile.

Ожидаемый результат:

- пользователь выбирает подрядчика, а не разбирается в чанках;
- карточки поддерживают сценарий `тема -> подрядчик -> документы -> фрагмент`.

## Этап 5. Улучшить источники и цитирование

Цель: повысить доверие к RAG-ответу.

Изменения:

- добавить snippet найденного фрагмента в `SourceList`;
- сделать маркеры `S1`, `S2`, `S3` кликабельными;
- связать inline citations в ответе с блоком источников;
- при возможности вести источник к документу или конкретному фрагменту;
- показывать проблему данных для неизвестных страниц вместо `?`.

Ожидаемый результат:

- пользователь быстро проверяет, на чем основан ответ;
- ARGUS выглядит как проверяемый аналитический инструмент, а не как генератор текста.

## Этап 6. После UI улучшить качество retrieval

Цель: повысить качество поиска без перегрузки интерфейса.

Backend-зоны:

- `backend/app/features/search/use_cases/answer_global.py`
- `backend/app/features/search/use_cases/rag_common.py`
- `backend/app/features/search/ports.py`

Изменения для рассмотрения:

- включить reranker, когда будет готова инфраструктура;
- добавить hybrid search: dense vectors + sparse/BM25-like поиск;
- лучше обрабатывать точные сущности: ИНН, суммы, даты, номера договоров;
- отдельно ранжировать подрядчика, документ и конкретный фрагмент;
- не отдавать frontend технический score как главный пользовательский сигнал.

Ожидаемый результат:

- поиск лучше находит смысловые совпадения и точные договорные признаки;
- frontend получает уже нормализованные сигналы для отображения.

## Приоритет на первый sprint

Минимальный набор изменений с максимальным UX-эффектом:

1. Убрать дубль правой панели и источников.
2. Перенести поле поиска наверх.
3. Скрыть raw score.
4. Исправить markdown в ответе.
5. Привести `стр. ?` к нормальному состоянию.
6. Сделать карточки подрядчиков основным результатом.
7. После этого переходить к улучшению retrieval.

## Критерии готовности

Первый sprint можно считать завершенным, если:

- глобальный поиск отображается в один основной поток без постоянной правой зеркальной панели;
- поле запроса находится перед результатами;
- источники не дублируются;
- raw score не виден пользователю;
- markdown не протекает в текст ответа;
- неизвестные страницы не отображаются как `стр. ?`;
- карточки подрядчиков визуально являются главным результатом;
- экран работает на desktop и mobile без наложений и распирания длинными именами файлов.

# Интерфейс drill-down поиска

Интерфейс строится вокруг принципа **постепенного углубления в данные**:

> от общего поиска по базе → к подрядчику → к его договорам → к конкретным фрагментам договора.

Пользователь сначала формулирует общий запрос, например:\
**«найди поставщиков фруктов»**.

Система ищет релевантные фрагменты во всех загруженных документах, но в интерфейсе показывает не просто список чанков, а группирует результаты по подрядчикам. Это позволяет пользователю сразу увидеть, какие компании чаще всего связаны с нужной темой.

## Общая структура интерфейса

Интерфейс удобно реализовать в виде **трёхпанельного экрана**:

```
┌──────────────────────┬────────────────────────┬────────────────────────┐
│ Глобальный поиск     │ Карточка подрядчика     │ Карточка договора       │
│                      │                        │                        │
│ Запрос:              │ Подрядчик: ФруктТорг   │ Договор №456            │
│ "поставщики фруктов" │                        │                        │
│                      │ Все договоры:          │ Релевантные фрагменты:  │
│ Найденные подрядчики:│ - №456 от 2025         │ - стр. 3--4              │
│ - ФруктТорг          │ - №789 от 2024         │ - стр. 7--8              │
│ - ЯгодыЛТД           │ - №101 от 2023         │                        │
│ - Садовод            │                        │ Поиск внутри договора:  │
│                      │ Поиск по договорам:    │ "штрафы"               │
│                      │ "штрафы"              │                        │
└──────────────────────┴────────────────────────┴────────────────────────┘
```

## Левая панель: глобальный поиск

Левая панель отвечает за **поиск по всей базе документов**.

Пользователь вводит общий тематический запрос, например:

```
поставщики фруктов
```

Система выполняет семантический поиск по всем чанкам без дополнительных фильтров. Найденные чанки группируются по `contractor_entity_id`, то есть по сущности подрядчика.

В результате пользователь видит список подрядчиков, связанных с запросом:

```
ФруктТорг
3 релевантных фрагмента из 5 договоров

ЯгодыЛТД
2 релевантных фрагмента из 2 договоров

Садовод
1 релевантный фрагмент из 3 договоров
```

Главная задача этой панели --- помочь пользователю не читать отдельные фрагменты сразу, а сначала выбрать релевантного подрядчика.

## Средняя панель: карточка подрядчика

Когда пользователь выбирает подрядчика, открывается средняя панель.

Она показывает информацию по конкретной компании:

```
Подрядчик: ООО «ФруктТорг»

Договоры:
- Договор №456 от 15.01.2025
- Договор №789 от 03.08.2024
- Договор №101 от 21.11.2023
```

Здесь пользователь видит все договоры выбранного подрядчика. Их можно сортировать по дате, релевантности к исходному запросу или типу документа.

Также в этой панели можно добавить отдельный поиск по всем договорам подрядчика. Например:

```
штрафы
```

В этом случае система ищет только внутри документов выбранного подрядчика, используя фильтр:

```
{
  "contractor_entity_id": "c_123"
}
```

Так пользователь может быстро найти все упоминания штрафов, рисков, поставок, сроков или условий оплаты по конкретному подрядчику.

## Правая панель: карточка договора

Когда пользователь выбирает конкретный договор, открывается правая панель.

Она показывает содержимое и релевантные фрагменты выбранного документа:

```
Договор №456 от 15.01.2025

Релевантные фрагменты:
- стр. 3--4 --- условия поставки фруктов
- стр. 7--8 --- ответственность сторон
- стр. 12 --- порядок оплаты
```

Внутри договора пользователь может выполнить отдельный поиск, например:

```
штрафы за просрочку
```

Тогда система ищет только по чанкам выбранного договора, используя фильтр:

```
{
  "contract_id": "d_456"
}
```

Это удобно для точечного анализа конкретного документа: пользователь не теряется среди всех договоров и работает только с нужным контекстом.

## Основной пользовательский сценарий

Типовой сценарий выглядит так:

1. Пользователь вводит общий запрос:\
    **«поставщики фруктов»**

2. Система показывает список подрядчиков, связанных с этой темой.

3. Пользователь выбирает подрядчика:\
    **ООО «ФруктТорг»**

4. Система показывает все договоры этого подрядчика.

5. Пользователь выбирает договор:\
    **Договор №456**

6. Система показывает релевантные фрагменты договора.

7. Пользователь уточняет поиск внутри договора:\
    **«штрафы за просрочку»**

## Почему такой интерфейс удобен

Такой интерфейс не превращает поиск в простой список найденных фрагментов. Вместо этого он сохраняет бизнес-логику документов:

```
Тема → Подрядчик → Договор → Фрагмент
```

Пользователь постепенно сужает область поиска:

```
Вся база документов
↓
Документы конкретного подрядчика
↓
Один выбранный договор
↓
Конкретные страницы и фрагменты
```

Это особенно удобно для анализа большого количества договоров, потому что пользователь сначала видит общую картину, а затем может быстро перейти к деталям.
